# AI Capriccio

AI 正在改变我们工作的方式。  
但从“会用”，到“用好”，中间隔着很长一段路。

这里不是最佳实践手册，也不是某种确定无疑的方法论。  
更像是一份持续展开的工作笔记：记录我在推进 AI 工具、Agent 系统与工程实践时，遇到的问题、形成的判断，以及一些暂时还不成熟、但值得留下来的想法。

有些内容偏工程，有些偏认知，有些只是对当下 AI 工作方式的一点观察。  
它们未必完整，但希望真实。

## 我在这里写什么

大致会围绕三类问题：

### 1. AI 如何真正进入工作流
不是“模型能做什么”，而是“一个人、一个团队，怎么把 AI 真正变成工作方式的一部分”。

### 2. Agent 和工具系统怎么搭
包括 skill design、工具封装、workflow 编排、CLI/HTTP bridge、系统边界这些更偏工程的问题。

### 3. AI 时代的人的负担与判断
AI 会放大效率，也会放大并行、选择、噪音和疲惫。  
很多问题最后不是技术问题，而是认知负担、决策密度和授权方式的问题。

## 文章

### [AI 时代，也要给大脑回蓝](docs/brian-congitive.md)
当 AI 降低了启动成本，事情会越来越多、越来越碎。  
真正被消耗的，往往不是体力，而是决策密度和注意力蓝条。

### [白嫖 Claude API](docs/cli-as-api.md)
一个偏工程实践的思路：把 Claude Code CLI 作为可调度后端，用子进程模式接出一层 HTTP 接口。  
重点不是“省钱技巧”，而是重新理解 CLI、Agent 和 API 的边界。

### [Vibe Fantasy or Fallacy: Prompt As Algorithm](docs/vibe-fantasy.md)
vibe coding 很容易让人高估 prompt，低估系统。  
当确定性逻辑被“模型化”之后，很多问题不会立刻爆炸，但迟早会。

### [Agent Skill 设计模式](docs/agent-skills.md)
围绕 Google ADK SkillToolset 的几个模式做了整理，并补了几个更偏系统架构层的模式。  
适合从“单个 Agent”往“Agent 平台”过渡时看。

## 这个仓库为什么放在 GitHub

可能最直接的原因，确实是 version control。  
但更重要的是：很多关于 AI 的想法，不适合只停留在聊天窗口里。

写成文档、留下版本、不断修订，才能慢慢看清楚哪些只是情绪，哪些才是判断。
