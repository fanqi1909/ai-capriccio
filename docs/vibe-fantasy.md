# Vibe Fantasy or Fallacy: Prompt As Algorithm

最近看了不少 vibe coding 的项目，自己也做了一些，有点感想。

## 工程师 vs 产品经理的 Vibe 方式

我观察到一个现象：产品经理和工程师 vibe 出来的东西，形态完全不一样。

产品经理习惯从用户视角出发，vibe 的是 **user flow**——"用户有什么痛点，AI 能帮用户做什么"。所以他们的产出大多是单点工具：TG 情报监听、技术指标分析、交易计划生成、复盘报告。本质上是"人驱动 AI"——人提问，AI 回答，人决策，人执行。AI 是个高级计算器。

工程师想的不一样。工程师 vibe 的是 **架构**——"这个系统要自己跑，状态怎么管、调度怎么做、边界在哪"。所以工程师的产出往往是 infra prototype：agent 之间怎么通信、状态怎么隔离、trust boundary 在哪、怎么做可观测性。

两种思维都有价值，但 vibe coding 对这两种人的效果完全不同：

- 产品经理 vibe 出来的是 **feature demo**，证明"这个场景 AI 能做"
- 工程师 vibe 出来的是 **infra prototype**，证明"这套架构能支撑 AI 自主跑"

不过这篇文章不是想说谁好谁坏。我想聊的是 vibe coding 过程中一些隐蔽的坑——这些坑 AI 不会主动告诉你。

## Case 1: 确定性逻辑被模型化

用户说"帮我每小时检查行情，符合条件就自动开仓"，AI 会 vibe 出这样的东西：

```markdown
# SKILL.md
每小时检查 BTC 行情，如果 RSI < 30 且价格在支撑位附近，就开多单。
```

看起来很简洁，对吧？

问题是：
- **RSI < 30** 这个判断，模型每次算可能有微小误差
- **"支撑位附近"** 这个定义本身就模糊，模型每次理解可能不一样
- 同样的行情，跑两次可能得出不同结论
- 出了问题，你根本不知道是哪个环节的判断出了偏差

工程师会怎么做？

```python
# 确定性逻辑，代码写死
def should_open_long(candles):
    rsi = calculate_rsi(candles, 14)
    support = find_support_level(candles)
    price = candles[-1]['close']
    return rsi < 30 and price < support * 1.02

# 模型只负责不确定的部分
def get_sentiment(news):
    return llm.analyze(news)  # 这个可以模糊
```

**确定性的逻辑用代码，不确定的部分用模型。** 但 vibe coding 的用户分不清这个边界，一股脑全丢给模型。模型很乐意接，因为它什么都能"做"。但"能做"不等于"做得稳"。

## Case 2: 边际价值递减 vs 线性成本

我看到一个数据处理的 vibe 项目：模型持续调用 web search，然后用模型打标，模型分类聚合，最后产出一个很漂亮的报表。

表面上看，流程跑通了，报表很漂亮，用户很满意。

但仔细想想这个成本结构：

```
第 1 条数据   → 信息量 100%  → 花 1 token
第 10 条数据  → 增量信息 10% → 花 1 token
第 100 条数据 → 增量信息 1%  → 花 1 token
第 1000 条数据 → 增量信息 0.1% → 花 1 token
```

聚合场景下，数据越多，每条数据的边际贡献越小，但模型处理每条的成本是恒定的。**你花了 1000 token 去获取那 0.1% 的增量信息。**

用模型处理数据的 token 消耗，直觉上远大于数据本身的价值。

工程师会怎么做？
- **采样**：1000 条数据，随机抽 100 条让模型处理，统计置信区间足够就停
- **分层**：先用规则/小模型粗分类，只有"不确定"的才丢给大模型
- **聚合前置**：先用 SQL/代码做确定性聚合，模型只处理聚合后的 summary

但 vibe coding 的用户不会这样想。Demo 阶段数据量小，成本不敏感。AI 写代码的时候也不会主动提"这个很贵"。报表很漂亮，感觉"值了"——直到月底看到账单。

## Case 3: Prompt 伪装成算法

这个是最隐蔽的。

我自己做了一个项目：把一堆文章按主题分类。模型给了我个方案，我没细看就试了。方案里有一句：

> "当两个对话的主题相似时，请合并这两个主题"

跑起来我发现很不稳定——有时候一堆文章生成 2 个主题，有时候生成 3 个。

调了很久才发现问题：**这句话看起来是个指令，实际上是个聚类算法。** 但模型没有告诉我：
- "相似"的阈值是多少？
- 怎么度量"相似"？
- 合并的顺序是什么？
- 什么时候停止合并？

这些全靠模型每次跑的时候"自由发挥"。

工程师会怎么做？

```python
# 1. embedding 向量化（确定性）
embeddings = [embed(article) for article in articles]

# 2. 聚类算法（确定性）
clusters = hdbscan.cluster(embeddings, min_cluster_size=5)

# 3. 模型只负责起名字（可以模糊）
for cluster in clusters:
    cluster.name = llm.summarize(cluster.samples[:3])
```

同样的输入，永远得到同样数量的 cluster。模型只干它擅长的事：给东西起名字。

**这个 case 的隐蔽性在于**：模型用 prompt 伪装了算法。代码能跑，结果看起来也合理。你需要跑很多次、对比结果，才会意识到"等等，为什么不一样"。而且你跟 AI 调试的时候，AI 也不会告诉你问题出在哪——因为它自己就是问题的来源。

还有一个隐藏的成本问题：prompt 作为"算法"，复杂度直接爆炸。

```
确定性算法：
- embedding: O(n)
- 聚类: CPU 计算，约等于免费

Prompt 作为算法：
- n 篇文章两两比较: O(n²) 次调用
- 每次调用都带着那段 prompt
```

1000 篇文章，确定性方案是 1000 次 embedding 调用。Prompt 方案最坏情况是 100 万次模型调用，每次还重复发送那段 prompt。

**Prompt 是 O(1) 写一次，但 O(n²) 付费。**

## 为什么 AI 不会帮你发现这些问题？

1. **AI 的思维方式是 prompt-native** —— 它自己就是靠 prompt 工作的，所以它给的方案天然偏向"用 prompt 解决一切"

2. **AI 优化的是"需求被满足"，不是"成本最优"或"稳定性最高"** —— 你说要聚类，它给你聚类，不会说"这个方案太贵"或"这个方案不稳定"

3. **AI 没有被生产环境坑过** —— 它没有"这样写会出事"的直觉，没有工程师那种被坑出来的 checklist

所以 vibe coding + AI 是一个正反馈循环：用户不懂，AI 也不提醒，一路 vibe 下去，直到撞墙。

---

## So, Fantasy or Fallacy?

回到标题：**Prompt As Algorithm**，到底是 fantasy 还是 fallacy？

说它是 **fantasy**，因为它确实很诱人——用自然语言描述逻辑，不用写代码，不用懂算法，AI 就能帮你实现。Demo 跑通的那一刻，你会觉得"未来已来"。

说它是 **fallacy**，因为它在骗你——prompt 看起来像算法，但它不是。它是"建议"，不是"指令"。模型可以听，可以不听，或者听一半。你以为你在编程，其实你在许愿。

**Vibe coding 的陷阱不是跑不通，是跑通了但你不知道哪里是软的。**

Prompt 能做很多事，但它不该做所有事。知道什么时候该用 prompt、什么时候该用代码——这个判断力，AI 给不了你。

至少现在还给不了。
