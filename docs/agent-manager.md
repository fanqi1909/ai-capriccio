# Agent Managers 的时代

*2026年3月 · Retail Trading Claude AI Day 开场 Keynote · Jay Fan*

---

> 我本来给今天这个活动起名叫 Workshop。但后来我改主意了，改成了 Conference。
>
> Workshop 是我教你、你跟我学的逻辑。但我看到大家的内容，明明已经是百家争鸣的状态了。你今天来这里，不是来学习的，是来找共鸣、找应征的。

---

## 一、我们已经在实战，不是在观望

我们团队这几个月，已经在实战 AI 了。不是说"要不要用"——是已经用上了，而且用得挺深的。每个人有自己的探索路径，踩过坑，总结过经验，有自己的一套心得。

你有自己的体会，说出来，看看有没有人跟你想到一块儿去——这才是今天最有价值的事情。

---

## 二、AI 对行业的冲击：从千人一面，到千人千面

我女儿今年6岁。上周，她坐在我旁边，跟 Claude 对话，1个小时做了一个 iPad app。

她不会写代码，什么都不会。但她知道自己想要什么——这里加个按钮，那里换个颜色，这个动画要快一点。她就说，Claude 就做。做出来的东西，完全是她自己的想法，跟市面上任何一个 app 都不一样。

以前做一个 app 要什么？产品经理写需求，设计师出稿，工程师开发，几个月才能上线。现在呢？一个6岁的小孩，1个小时。

**创造定制化体验的门槛，正在消失。**

以前软件是千人一面的——一个 app 服务所有人，大家用的是同一套界面，同一套逻辑。为什么？因为做定制化太贵了，成本摊不开。现在这个逻辑变了。每个人都可以拥有专属于自己的界面、自己的交互方式、自己的体验。

这对产品意味着什么？标准化的 UI 方案，价值在重构。真正有价值的，是核心的业务逻辑、细粒度的 API——因为那是 AI 可以灵活调用、灵活组合的东西。

---

## 三、AI 对个人的冲击：新的竞争维度

LLM 把所有的 Knowledge Skill 都拉平了。

以前，你的竞争力来自你知道的比别人多。你学了5年的系统设计，你熟悉某个框架的底层原理——这些让你在团队里有不可替代的价值。现在呢？你问 Claude，它什么都知道一点。那个"你知道、别人不知道"的优势，被大幅压缩了。

但这不代表你的价值消失了。它只是换了一个维度。

**新的竞争维度：**

- 不是你**知道多少**
- 而是你知道你有多少**不知道的**——你的元认知，你清不清楚自己的盲区在哪里
- 甚至是你知道有多少 **LLM 不知道的**——你对业务上下文、对 production 细节的理解，LLM 根本不知道

这是一个对所有人公平的新竞争维度。它不取决于你的技术背景有多深，而取决于你的思维有多清晰、你对问题的理解有多准确。

## 四、言值即正义

> 你说得越清楚，它干得越好。**表达能力，就是生产力。**

这不是比喻——是字面意思。

在 AI 时代，你说得越准确、越清晰、越有结构，它干得越好。你说得含糊、模棱两可，它给你的也是一堆含糊的东西。

这一点，对大家是公平的。你不需要会写 Python，不需要懂 Kubernetes，你只需要能把你脑子里的想法，清晰地表达出来。如果你在想"我接下来要培养什么能力"，我的答案是：练习表达。把想法说清楚的能力，在 AI 时代会越来越值钱。

---

## 五、会不会砍人？不会

> AI 是加速器，不是替代器。

我知道大家心里都有这个问题。我来直接回答：不会。

我们的 MCP 项目，前期的设计加代码，两天做完了。以前这个事情要两周。那这件事就结束了吗？不是的。代码做完才是起点——做完之后，要不要宣发？要怎么让更多人知道？要不要观察生态的变化？要不要去搞运营，找合作？要不要优化上下文，让 agent 跑得更稳？

这些问题，没有一个是 AI 能替你决策的。

所以逻辑不是"以前2个人干两周，现在1个人干两天，砍掉1个人"——逻辑是那个"省出来的人"，去扩展下一个边界。

**AI 帮你加速到一个方向的边界。边界之后怎么突破——是人的事。**

反过来说，对人的挑战其实变大了。AI 把你推得更快，但方向盘还在你手里。你要做的决策更多、更复杂，因为你更快到达那个需要判断的地方了。

## 六、AI 没有 Taste

AI 没有 Skin in the Game，所以它没有 Taste。

更准确地说，AI 不知道"能跑"和"对的"之间的区别。你让它解一道题，它给你一个答案。但它不知道这个答案在你的业务场景下是不是合理的，它不知道这个设计方案有没有潜在的风险，它不知道这个 API 的命名在你们团队里有没有歧义。

根本原因是：它没有 skin in the game。你会被 oncall，你的名字在 commit 里，你要为 production 出了问题负责。AI 呢？它没有任何后果，所以它没有真正意义上的 taste。

你经历过的那些 bug，你半夜被 pager 叫醒的焦虑，你在 code review 里被喷过的那些决策——那些经历，逐渐积累成了你的 taste。那个"这里感觉不对"的直觉，就是你的核心价值，AI 取代不了。

> 人的价值，从"能不能做"，转移到了"**值不值得做**、**怎么做才对**"。

AI 会把你的判断放大十倍执行。判断准了，它帮你把优势放大十倍。判断偏了，它帮你把偏差也放大十倍。所以你的判断力，从来没有像现在这么重要过。

## 七、应对：每个人都晋升了

今年，每个人都晋升了。以后大家都是 **Agent Manager**。

以前，我们是工程师，我们写代码。现在，我们有了 agent 帮我们执行。那我们是什么？我们是 manager。

Agent Manager 要做三件事：

**01 Context Engineering**

Repo 是最好的知识库。这句话的意思是——agent 在运行时，能接触到的信息，才是它能用的信息。与其花大量精力搭一个企业知识库，不如把知识直接沉淀进 repo。代码注释、架构决策文档、API 说明——这些放进 repo，agent 就能用到。

**02 Architectural Constraints**

给 agent 设护栏，不是靠每次 prompt 里写"注意别做这个"——那太脆了，每次都要说，迟早漏掉。真正稳定的护栏，是用 linter、结构测试、CI 检查来约束。让 agent 物理上就在正确的轨道上跑。

**03 Garbage Collection**

系统会自然腐化。文档会过时，架构会漂移，规范会被违反。这不是谁的错——这是熵增，是系统的自然状态。Agent Manager 的工作，就是定期运行 agent，扫描哪些文档过时了，哪些地方架构违规了，把这些问题清理掉。对抗熵增，是一个长期的、持续的工作。

这三件事合在一起，有个更精确的名字：**Harness Engineering**。Harness 是一套约束、工具、文档和反馈循环的集合，让 AI agent 保持高效和正轨。Context Engineering、Architectural Constraints、Garbage Collection，正是 Harness 的三大支柱。

---

## 八、Agent 出错 = 系统信号

> Agent 出错，不是 agent 的错——是系统信号。

当 agent 跑偏了，大家的第一反应往往是去调 prompt，加更多约束，说"下次别这样"。这是没用的。

正确的反应是：去找缺失的东西。是不是缺了一个工具？是不是缺了一条护栏？是不是某个文档没写清楚？找到根本原因，反馈进 repo，让系统下次不会再出同样的问题。

Martin Fowler 最近说了一句话：

> "工程师的主要工作不再是写代码，而是设计环境、明确意图、构建反馈循环，让 agent 可靠地工作。"

这，就是 Agent Manager 要做的事。
